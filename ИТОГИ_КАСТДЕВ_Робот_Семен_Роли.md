### Сводный обзор (обновлено 2025-11-07)

- **Роли и ответственность**: подбор кодов (ОКПД‑2/КТРУ) — зона ответственности закупщика (`buyer`). Инициатора не учим кодам: он заполняет короткую, понятную форму (3–5 вопросов) и может приложить фото/модель. Закупщик быстро подтверждает/корректирует.
- **Агрегированные боли**:
  - Документы/согласования/КЭП: узкое горлышко, длинные циклы.
  - Интеграции/ЕИС: нестабильность, расхождения ОКПД‑2 → риски оплаты.
  - ТЗ: избыточная детализация/переквалификация; нет единых шаблонов.
  - НМЦК: ручной сбор КП и обоснований; низкий ответ поставщиков.
  - Коды КТРУ/ОКПД‑2: сложно даже для экспертов; у инициаторов — «мутные» заявки.
  - Организация процесса: децентрализация, путаница статусов «заявка vs закупка».
- **Ключевые решения продукта**:
  - ИИ‑подбор кодов с объяснимостью и альтернативами.
  - Категорийные мастера (первый приоритет: Авто, Освещение/лампы, ИБП, Оргтехника).
  - Короткая форма для инициатора + загрузка фото/модели.
  - Пакетная обработка 100–150 позиций (xlsx/doc) с точечными уточнениями.
  - «Анти‑ошибка ЕИС»: предиктивная проверка расхождений кодов между системами.
  - НМЦК‑помощник: запрос КП, расчёт, региональные фильтры, алерты на неполноту.
  - Командный режим (веб): роли/права, уведомления в Telegram/email, фильтры по отделам.
- **Метрики успеха**: t_intake (разбор «мутных» заявок), t_pick_code (подбор кода у закупщика), first‑pass intake, число/доля расхождений ОКПД‑2↔ЕИС, время «описание→НМЦК», NPS закупщика.
- **Валидация спроса**: театр (готовность «заставить» инициаторов тестировать), медучреждение (~6 закупщиков) — выбор веб‑теста, позитивная реакция на ИИ‑бота.
- **Следующие шаги**: пилот 2–3 недели (50–100 позиций), приоритизировать мастера категорий, онбординг отдела (веб), сбор «до/после» по метрикам.

### Краткая сводка (обновлено 2025-10-22 15:25:00)
- **документы/подпись КЭП/ЭЦП**: главный узкий горлышко — согласования, подписи руководства, сбор сканов.
- **интеграции и площадки (ЕИС/РТС/Сбер/региональные РИС/ГИС)**: нестабильности, ручные переносы, дублирование шагов.
- **перегруз ТЗ**: даже для простых позиций чрезмерная детализация; запрос на шаблоны и упрощение.
- **НМЦК**: ручной сбор цен/КП, обоснование требует автоматизации.
- **поставщики**: ориентируются на уже известных; поиск новых затруднён.
- **контракты и сроки**: ожидается рост контрактов к концу сентября — критичны статусы и контроль дедлайнов.
- **боли инициаторов**: отдел госзаказа "шлёт куда подальше" с подбором КТРУ; рядовые сотрудники тратят "безумное количество времени" на поиск кодов; нужен помощник для автоматизации подбора кодов и ТЗ.
- **боли энергетиков/технических специалистов**: каждый сотрудник каждого отдела сам готовит ТЗ; "все это ерундой приходится заниматься вынужденно"; нужен бот для автоматизации: "закидываете хочу купить монитор, и он вам сразу и код подбирает, и характеристики".
- **боли специалистов по закупкам**: проблемы с интеграциями ЕИС (ОКПД-2 не совпадает между системами); "искусственный интеллект еще не совершенен"; сложности с техподдержкой ЕИС; децентрализация работы в разных городах.
- **боли инициаторов (технические специалисты)**: "не по своей должности" заниматься закупками; отдел закупок "не справляется" с объемом; нужна помощь в поиске кодов и ТЗ.

### Риски для клиентского опыта
- **срывы дедлайнов** из‑за подписей/сканов и нестабильных интеграций.
- **ошибки переноса данных между системами** → жалобы/отклонения.
- **переквалификация требований ТЗ** → снижение конкуренции/жалобы ФАС.
- **низкая прозрачность статусов оплаты/исполнения**.
- **фрустрация инициаторов** из‑за "безумного количества времени" на поиск кодов КТРУ и отсутствия помощи от отдела госзаказа.
- **фрустрация технических специалистов** из‑за необходимости "заниматься ерундой вынужденно" вместо основной работы; каждый отдел сам готовит ТЗ без единых стандартов.
- **критические ошибки интеграций ЕИС** (ОКПД-2 не совпадает между системами) → невозможность оплаты контрактов → "красная зона" просрочек.
- **неэффективность техподдержки ЕИС** → длительное решение проблем через переписку вместо звонков.

### Быстрые возможности (MVP-гипотезы)
- **умные шаблоны ТЗ**: авто‑проверки, подсветка избыточной детализации, типовые поля.
- **трекер подписи/КЭП**: статусы, напоминания, ответственный «кто держит мяч», SLA по согласованию.
- **НМЦК‑помощник**: сбор ≥3 цен из источников, авто‑обоснование в стандартизованный шаблон.
- **монитор интеграций (ЕИС/ЭТП)**: алерты, авто‑повторы, журнал расхождений с возможностью ручной сверки.
- **поиск/рекомендации поставщиков**: по региону/номенклатуре за один клик.
- **единый статус‑скрин**: от план‑графика до оплаты/исполнения с ключевыми датами и рисками.
- **КТРУ‑бот для инициаторов**: автоматический подбор кодов по описанию товара/услуги + генерация ТЗ в структурированном виде (приоритетная гипотеза на основе транскрибаций).
- **ТЗ‑бот для технических специалистов**: "закидываете хочу купить монитор, и он вам сразу и код подбирает, и характеристики" — автоматизация подготовки технических заданий (валидированная гипотеза).
- **монитор ошибок ЕИС**: автоматическое отслеживание несоответствий ОКПД-2 между системами + алерты + рекомендации по исправлению.
- **помощник по поиску ТЗ**: "поиск документации" по наименованию товара с готовыми техническими заданиями (валидированная гипотеза).
